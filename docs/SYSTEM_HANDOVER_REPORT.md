# تقرير تسليم النظام التقني الشامل (System Handover Report)

## 1. ملخص المشروع (Project Overview)
المنصة عبارة عن نظام SaaS (Software as a Service) متعدد المستأجرين (Multi-tenant) لإدارة أتمتة الواتساب والمحادثات للشركات. تم تصميم النظام ليقوم بدور الوسيط الذكي الذي يتيح للعملاء إعداد ردود آلية، إرسال حملات (Broadcasts)، إدارة جهات الاتصال، وتتبع رحلة العميل دون الحاجة للتدخل في التعقيدات البرمجية لأدوات مثل n8n. كل ما يتعلق بالبنية الأساسية للأتمتة يتم إدارته من قبل مدير المنصة (Super Admin).

## 2. التقنيات المستخدمة (Tech Stack)
* **الواجهة الأمامية (Frontend):** React.js باستخدام Vite.
* **لغة البرمجة والتكويد:** TypeScript مع تفعيل فحص الأنواع (Strict Type Checking).
* **تنسيق الواجهات (Styling):** Tailwind CSS لاعتماد تصميم مستجيب وعصري.
* **الحركات والانتقالات (Animations):** Framer Motion / `motion/react` لعمل انتقالات ناعمة وحالات تحميل متقدمة (Skeleton Loaders).
* **الأيقونات:** Lucide React.
* **قواعد البيانات والمصادقة (Backend/BaaS):** Firebase (Firestore كمستودع للبيانات، Authentication عبر Google لتسجيل الدخول).

## 3. البنية المعمارية (Architecture Pattern) - SaaS & n8n
تم تصميم وتوثيق مسار العمل التقني (موجود مسبقاً في `docs/ARCHITECTURE.md`) بناءً على المعايير التالية:
* **عزل بيانات العملاء (Multi-Tenancy):** استخدام `orgId` في جميع العمليات لعزل بيانات كل شركة عن الأخرى.
* **إخفاء تعقيد n8n:** يتم إنشاء "مسارات عمل رئيسية" (Master Workflows) داخل n8n لجميع الشركات. واجهة المنصة تقوم فقط بحفظ الإعدادات والمحفزات، وعند حدوث حدث ما، يرسل النظام Webhook إلى n8n محمل بـ `tenant_id` وقواعد الذكاء الاصطناعي الخاصة بالشركة، ليقوم n8n بالرد بالنيابة عنها.
* **الفصل بين الصلاحيات:**
  - لوحة تحكم `SuperAdmin`: لإدارة الـ Webhooks، الاشتراكات، تفعيل وتعطيل المؤسسات، ومراقبة الجودة.
  - لوحة تحكم `Dashboard`: لوحة العميل لإدارة المحادثات والقوالب وجهات الاتصال وبناء الأتمتة بصرياً.

## 4. المكونات والصفحات المنجزة (Completed Features & Modules)

### 1- المصادقة والتسجيل (Authentication)
* صفحة تسجيل الدخول (`Login.tsx`).
* مزود المصادقة `AuthProvider.tsx` لتتبع حالة المستخدم الحالية (Session) وتوجيهه للمسار الصحيح.

### 2- لوحة التحكم الرئيسية (Dashboard Home)
* ملخص الأداء والإحصائيات.
* تفاصيل آخر الحملات (مكتملة / نشطة).
* استخدام `AnimatePresence` وحالات تحميل هيكلية (Skeletons) لانتقال ناعم للبيانات.

### 3- صندوق الوارد (Inbox)
* واجهة محادثات حية مقسمة لـ 3 أعمدة (قائمة المحادثات، الشات، تفاصيل العميل).
* فلاتر متقدمة (متصل الآن، غير مقروء، رسائل أتمتة نشطة).
* نافذة معلومات العميل الجانبية تعرض (الوسوم، ملاحظات الأتمتة، ورحلة العميل).

### 4- إدارة العملاء وجهات الاتصال (Contacts)
* جدول إدارة جهات الاتصال وتصنيفها بوسوم (Tags).
* زر لاستيراد وتصدير العملاء (تم تفعيل **تصدير العملاء كـ CSV** مع جلب كافة التفاصيل).
* عرض مفصل **لرحلة العميل (Customer Journey)** يظهر تسلسل زمني (Timeline) متى تم الرد الآلي، متى اشترى، ومتى تحول لمتحدث بشري.

### 5- الأتمتة والبناء البصري (Automations)
* قائمة لعرض الأتمتات الحالية وبيانات واضحة عن (تردد التفعيل، نسبة النجاح، والحالة).
* نموذج أولي للباني البصري (Builder View) لتكوين تسلسل العمليات بأسلوب مبسط للعميل.

### 6- البث وحملات الرسائل (Broadcasts)
* صفحة لإنشاء وإرسال حملة بث جديدة (Modal).
* عرض تاريخ ومقاييس الحملات القديمة (كم شخص فتح الرسالة، كم تم الإرسال).

### 7- الرسائل الجاهزة (Templates)
* إدارة القوالب (Templates) مثل قوالب التسويق، الإشعارات، ورمز التحقق (OTP).
* القدرة على نسخ القالب لطلب اعتماده مباشرة من Meta.

### 8- الإعدادات (Settings)
* تبويبات منسقة: (المعلومات الشخصية، الأمان [2FA] والجلسات النشطة، الفواتير والاشتراكات لترقية الباقة، ارتباط واتساب Business API، دعوة فريق العمل).

### 9- لوحة المشرف العام (Super Admin)
* مخصصة لحسابات الـ Super Admins.
* إدارة المؤسسات (Tenants)، إيقاف تفعيل الشركات، وإدارة المستخدمين.

## 5. هيكل بيانات قواعد البيانات (Firebase Firestore Schema Concept)
النظام يعتمد على الهيكل الدلالي التالي (قيد استكمال التطبيق الميداني الفعلي):
* **`organizations/`**: يحتوي التكوين العام للشركة (planId, whatsappConnected, n8nWebhookUrl, usageMetrics).
* **`users/`**: يحمل معلومات المستخدم و `orgId` و `role` (سواء owner, admin, agent).
* **`contacts/`**: جهات الاتصال التابعة لكل مؤسسة (مفلترة بـ `orgId`).
* **`chats/` & `messages/`**: المحادثات الجارية.
* **`automations/`**: قواعد سير العمل المنشأة من العميل.

## 6. المطلوب من المطور التالي / الأداة القادمة (Next Steps to Handover)
لأجل إكمال المشروع بالشكل الصحيح واستلامه عبر أداة ثانية، هذا ما يجب التركيز عليه:

1. **الربط الفعلي للمسارات (Backend Integration):** 
   - إنشاء خطافات الشبكة (Cloud Functions / Webhooks) للتنصت على رسائل واتساب والمزامنة الفعلية مع `n8n`.
2. **استبدال Mock Data بيانات وهمية ببيانات فعلية:**
   - حاليا، صفحات `Contacts` و `Inbox` و `Dashboard` تعرض بيانات وهمية لتسريع اعتماد واجهة المستخدم (UI/UX). يجب تطبيق الدوال في `services/` لجلبها فعلياً من Firestore.
3. **التكامل مع N8N Server:**
   - إعداد بيئة N8N وإنشاء سير العمل الرئيسي المستهدف في (ARCHITECTURE.md).
   - توجيه الـ API من وإلى واجهة المتجر الحالية لتفعيل المحادثات.
4. **تأمين Firestore (Security Rules):**
   - إضافة قيود صارمة على كل Collection ليضمن عدم تداخل معلومات منظمة (Organization) مع استعلامات منظمة أخرى.
